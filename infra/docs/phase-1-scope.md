# Phase 1 Scope

## 목표

Phase 1의 목표는 MODi MVP에서 보여줄 최소 사용자 여정과 데이터 범위를 고정하는 것이다.

이번 단계에서는 의료데이터 production system을 만들지 않는다. 대신 Sui, Walrus, Seal을 이용해 다음 흐름이 실제로 가능하다는 것을 증명한다.

1. 연구기관이 데이터 요청과 보상 조건을 등록한다.
2. 사용자가 요청을 확인하고 참여 여부를 결정한다.
3. 사용자 데이터는 원본 형태로 플랫폼에 저장되지 않는다.
4. 보호 처리된 데이터만 암호화되어 Walrus에 저장된다.
5. Sui에는 데이터 참조, 동의, 접근, 보상, 감사 이력만 남는다.
6. Seal은 유효한 동의와 접근 권한이 있을 때만 복호화를 허용한다.
7. AI Agent는 처리 결과가 정책을 만족하는지 검증하고, 사용자 선호와 이전 결정을 memory로 유지한다.

## 데모 시나리오

### 시나리오 이름

당뇨 연구용 가명처리 건강 데이터 제공

### 참여자

| 참여자 | 설명 |
|---|---|
| 연구기관 | 당뇨 관련 연구를 위해 30대 사용자의 가명처리 건강 데이터를 요청한다. |
| 사용자 | 자신의 데이터 제공 여부와 동의 범위를 직접 승인한다. |
| AI Agent | 데이터 처리 결과를 검증하고, manifest와 processing receipt를 생성 또는 확인한다. |
| 플랫폼 | 원본 데이터 없이 요청 목록, 동의 상태, 접근 상태, 보상 상태를 보여준다. |

### 최소 흐름

1. 연구기관이 Sui에 `DataRequest`를 생성한다.
2. 연구기관이 보상 금액을 `RewardEscrow`에 예치한다.
3. 사용자가 요청 조건을 확인한다.
4. 사용자 앱이 합성 건강 데이터를 준비한다.
5. 전처리 모듈이 금지 필드를 제거하고 데이터를 범주화한다.
6. AI Agent가 처리 결과를 검증한다.
7. 데이터와 manifest를 Seal로 암호화한다.
8. 암호문을 Walrus에 업로드한다.
9. 사용자가 Sui에 `DataAsset`과 `ConsentGrant`를 등록한다.
10. 연구기관이 `AccessGrant`를 생성한다.
11. 연구기관이 Walrus blob을 조회하고 Seal policy를 통해 복호화한다.
12. 접근 이력과 보상 지급 이력이 Sui event로 남는다.

## MVP 데이터 범위

MVP에서는 원본 PHI 대신 합성 JSON 데이터를 사용한다.

허용 필드는 다음과 같다.

| 필드 | 예시 | 설명 |
|---|---|---|
| `ageRange` | `"30-39"` | 정확한 생년월일 대신 연령대만 사용 |
| `gender` | `"female"` | 선택형 범주 |
| `regionCode` | `"KR-SEOUL"` | 상세 주소가 아닌 넓은 지역 코드 |
| `conditionTags` | `["diabetes"]` | 진단명 전문이 아닌 연구용 태그 |
| `glucoseRange` | `"140-179"` | 정확한 수치 대신 구간 |
| `deviceType` | `"wearable"` | 데이터 출처 범주 |
| `recordedMonth` | `"2026-05"` | 정확한 일시 대신 월 단위 |

## 제외 범위

Phase 1과 Phase 2에서는 다음을 구현하지 않는다.

- 실제 병원 EMR 연동
- 실제 PHI 저장
- HIPAA/GDPR/개인정보보호법 적합성 인증
- 복잡한 AI 모델 학습
- 다중 사용자 대규모 정산
- FHE 기반 집계 계산
- Walrus Sites 기반 프론트엔드 배포

## 성공 기준

Phase 1은 다음 산출물이 준비되면 완료로 본다.

- 데모 시나리오가 하나로 고정되어 있다.
- MVP에서 허용하는 샘플 데이터 schema가 정의되어 있다.
- 금지 필드와 처리 정책이 문서화되어 있다.
- AI Agent의 책임 범위가 정의되어 있다.
